# 产品需求文档（PRD）—— whodunit / 《雨夜书房》AI 剧本杀

- 日期：2026-06-08
- 关系：本 PRD 讲 **要做什么、给谁、达到什么标准**（产品视角）；**怎么实现**见设计文档 `2026-06-05-whodunit-design.md`。两者冲突时，技术细节以设计文档为准，产品意图以本文为准。

---

## 1. 一句话 & 愿景

一句话：**单个真人 + 一桌 AI 玩家 + 一个 AI DM，用一个固定剧本，完整跑完一局剧本杀。**

愿景：证明"AI 能不能当好剧本杀里的玩家与 DM"——它们要会演、会藏秘密、会推理、会撒谎，却**不开天眼、不自爆**。做成一个既能放进作品集（证明 agent 工程能力），又能给剧本杀同好真正试玩、按反馈迭代的产品。

## 2. 目标用户

| 用户 | 诉求 | 本产品给什么 |
|---|---|---|
| **作者本人**（首要） | 用一个真实项目证明 agent/多智能体工程能力（面试） | 可讲清、经得起追问的"信息隔离"架构 + 可复现的 eval 数字 |
| **剧本杀爱好者朋友** | 想体验"和 AI 玩剧本杀好不好玩" | 一桌随时能开、不用凑人的 AI 牌局 |
| **同行/招聘方**（间接） | 评估作者的工程判断与落地能力 | 干净的引擎代码 + 测试 + writeup |

## 3. 要解决的问题 / 价值主张

剧本杀好玩，但**凑人难、约时间难、需要真人 DM**。AI 桌可以随时开局。难点也正是价值所在：让 LLM 在**信息不对称、需要欺骗与推理**的对抗场景里演得像样，而不是"全知全能、一眼看穿"。本项目把"用架构约束全知模型"这件最难的事做透，作为差异化。

## 4. 核心用户场景（User Stories）

- **US1**：作为玩家，我能加入一桌（1 真人 + N 个 AI），从开场一路玩到复盘揭真相，全程不崩。
- **US2**：作为玩家，AI 对手会**守住自己的身份/秘密/目标**——该藏的藏、该撒谎撒谎、该推理推理，**绝不暴露它不该知道的信息，凶手绝不自爆**。
- **US3**：作为玩家，我能在浏览器里**看发言、打字发言、投票**（Phase 2）。
- **US4**：作为玩家，投票后能看到 **DM 郑重复盘、揭晓凶手与破案链**。
- **US5**：作为作者，我能**全自动跑 K 局**（无真人）并得到一张"完成率/泄密拦截/延迟/不变量"的数字表，用来证明系统行为可复现。

## 5. 功能需求（FR，MVP）

| 编号 | 需求 | 验收要点 |
|---|---|---|
| **FR1** | 完整一局流程闭环（开场→自我介绍→搜证→讨论→投票→复盘） | 阶段顺序正确、不跳阶段、到上限强制推进 |
| **FR2** | 信息隔离硬保证 | 装进任何 AI prompt 的信息**绝不含未授权 `info_id`**；由 VisibilityGate 单测 100% 证明 |
| **FR3** | AI 玩家守身份/秘密/目标 | 成局 transcript 中凶手不自爆；发言层泄密率低于阈值（统计验收） |
| **FR4** | 投票指认 + 复盘揭真相 | 投票格式合法、计票正确；复盘只依据确定性真相文本、不编造新证据 |
| **FR5** | 人在环（真人作为参与者完整参与） | 真人与 AI 走同一管道；轮到真人时等其输入（Phase 2） |
| **FR6** | 一个自洽的固定剧本（《雨夜书房》） | 线索链闭合、真相唯一、障眼法成立 |

## 6. 非功能需求（NFR）

| 编号 | 需求 | 说明 |
|---|---|---|
| **NFR1 永不崩盘** | 任何单个 AI/LLM 失败只降级、不崩整局 | 重试→换模型→安全发言（design §7） |
| **NFR2 延迟可接受** | 一桌 AI 逐个调用不至于让人难等 | 流式 + 可并行阶段并行 + "思考中"态；spike 实测 ~6s/次 |
| **NFR3 成本可控** | 按角色路由模型（玩家→便宜量大，DM→更强） | 单局设调用预算上限 |
| **NFR4 可复现 eval** | 非确定性行为用数字而非"感觉"评估 | 录制回放 + 不变量断言 + 指标表 |
| **NFR5 隔离两层验收** | 不混为一谈 | prompt 组装层硬证明（确定性）+ 发言层统计达标（无法 100%） |

## 7. 范围 / 非范围（MVP）

详见设计文档 §2。要点：**做** 1 真人 + N AI、AI DM、1 个固定原创本、网页薄壳、本地运行；**不做** 多真人实时同步、真人 DM、AI 自动出本、精美 UI/语音、登录/合规。

## 8. 验收标准

1. 一局能从开场到复盘**不崩**地跑完（FR1 + NFR1）。
2. 信息隔离两层达标：prompt 组装层 100% 硬证明（FR2）；发言层泄密率低于阈值（FR3）。
3. eval 台产出硬指标：完成率、平均等待时长、泄密拦截次数、DM 卡阶段次数（NFR4）。
4. 体验信号（真北极星）：作者 + 几位剧本杀朋友试玩后反馈"AI 演得有点东西、推理还挺有意思"。

## 9. 里程碑

详见设计文档 §9（垂直切片分期）。摘要：
- **Phase 1**：硬核引擎 + eval 台（CLI，无前端）—— 当前进行中。
- **Phase 2**：人在环 + 第一版真实前端（验证好不好玩）。
- **Phase 3**：打磨成可分享的好产品（体验/视觉、第二个本、按反馈迭代）。

## 10. 关键风险

详见设计文档 §11。最致命三条：① AI 不肯演坏人/会自爆；② 推理体验天花板；③ 隐性信息泄露防不全。应对：游戏框架授权欺骗、选小而紧凑的本、闸门+规则+离线 judge 三道逼近。**spike 已验证 ①②③ 在可接受范围内。**
