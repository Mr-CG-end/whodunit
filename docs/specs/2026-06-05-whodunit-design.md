## 1. 目标与成功标准

**目标**
- 把"用架构约束一个本来全知的大模型、让它演好一个信息受限的角色"这件事吃透 —— 这是本项目的**核心差异化点**，也是作品集里最该讲清、最经得起追问的部分。
- 按真实开发标准，做出一个能真实试玩、可分享、可持续迭代的产品（而非一次性 demo），用来验证"AI 剧本杀好不好玩"。

**成功标准（MVP 验收）**
1. 一局能从开场到复盘**不崩**地跑完。
2. AI 玩家不偷看上帝视角、不泄露秘密 —— **分两层验收，不混为一谈**：
   - **prompt 组装层（硬验收，确定性可证明）**：装配进任何 AI 的 prompt 里，绝不含未授权的 `info_id`；由 VisibilityGate 单测 100% 保证。
   - **发言层（统计验收，无法 100% 证明）**：成局 transcript 经离线泄密样本集 / 语义 judge / 人工抽检，泄密率低于阈值。语义、暗示、同义改写型泄露只能逼近、不能根除（见风险 #3）。
3. 硬指标（由 §8 eval 台免费产出）：一局完成率、每轮平均等待时长、泄密拦截次数、DM 卡阶段次数。
4. 体验信号（学习 + 自娱项目的真北极星）：本人 + 几个剧本杀朋友各自试玩后，反馈"AI 演得有点东西、推理还挺有意思"。

## 2. 范围边界

| 维度 | ✅ MVP 做 | ❌ 暂不做（留到以后） |
|---|---|---|
| 玩家 | 1 真人 + N 个 AI 玩家（默认 5 人桌：1 真人 + 4 AI） | 多真人同桌、实时多人同步 |
| DM | AI DM 全程主持 | 真人 DM 模式 |
| 剧本 | 1 个原创、极简、自洽的固定剧本 | AI 自动出本 |
| 前端 | 薄壳：网页 H5（聊天式界面），打字输入 | 微信小程序、精美 UI、语音 |
| 账号/合规 | 无登录、本地运行、不对公众开放 | 算法备案、AI 内容标识、注册体系 |
| 其它 AI 角色 | 暂不做 | AI 创作助手、AI 新手引导助手 |

**为什么这样划**：学习优先 → 把精力集中在多智能体引擎；"单人 + 架构预留多人"是匠人式的选法（先把一件事做到极致，底子搭得能长大），同时避开"实时多人同步"这块与 agent 学习无关的大工程。

## 3. 整体架构（分层）

核心思想：**编排（流程状态图）与领域逻辑分离——TypeScript 主链路用手写的极简显式状态机推进流程；剧本杀特有的"记忆 + 信息隔离"自研。** 当前编排不依赖框架；若后续确实需要 checkpoint / 恢复 / 人工中断，再以独立实验评估框架适配，不改领域核心。

```
┌──────────────────────────────────────────────────────────┐
│ ① 前端薄壳 (H5 网页) —— 聊天式界面：看发言、打字输入、投票      │
└───────────────────────────┬──────────────────────────────┘
                            │ HTTP / SSE（Phase 2）；WebSocket 后置
┌───────────────────────────▼──────────────────────────────┐
│ ② 对局服务 (Node.js / TypeScript)                          │
│    含 MessageChannel 抽象 + Participant 抽象               │
│    (HumanParticipant / AIParticipant)                     │
└───────────────────────────┬──────────────────────────────┘
┌───────────────────────────▼──────────────────────────────┐
│ ③ 编排层 (手写确定性状态机) ← 自研主链路                   │
│    对局状态机：开场→自我介绍→搜证→讨论→投票→复盘            │
└───────────────────────────┬──────────────────────────────┘
┌───────────────────────────▼──────────────────────────────┐
│ ④ 剧本杀核心 (自研) ← 主战场                                │
│   DM 编排Agent · Player Agent×N · 可见性闸门 · 记忆管理器   │
│   剧本加载器 · 越权/泄密检测                                 │
└───────────────────────────┬──────────────────────────────┘
┌───────────────────────────▼──────────────────────────────┐
│ ⑤ LLM 路由层 (模型无关) —— 玩家→便宜强模型(DeepSeek)        │
│    DM→更强模型；可热插拔                                     │
└───────────────────────────────────────────────────────────┘
```

**关键设计意图**
- ③ 与 ④ 解耦：换剧本、换玩法时 ④ 改、③ 基本不动。**③ 主链路用 TypeScript 手写极简显式状态机（领域规则与编排运行时分离）；框架实验不进主链路，只有当人在环 / 恢复收益超过依赖成本时才评估替换 ③，而不动 ④（见 §10、plan）。**
- "预留多人/语音"的缝在 ②：`Participant` 把"真人/AI"统一抽象；`MessageChannel` 把传输（Phase 2 先用 HTTP / SSE，将来可换 WebSocket / 音频）与引擎解耦。
- **分期实现，别过早盖传输层**：上图是**目标形态**（前端 / 传输是 §9 的 Phase 2+ 才接）；Phase 1 先把 ②③④⑤ 的引擎与 eval 在 CLI 下做扎实，**不接前端 / WebSocket**。Phase 0 的 spike（已完成）只走 CLI、只实现 `Participant` 接口。多人 / 语音仅保留接口语义、不提前实现复杂通道（分期见 §9 修订）。

## 4. 核心组件（职责 / 接口 / 依赖）

| 组件 | 职责 | 接口（示意） | 依赖 |
|---|---|---|---|
| **ScenarioLoader** | 把剧本文件解析成结构化 `Scenario`（角色、各人私密剧本/秘密/目标、线索库+发放时机、真相/凶手、流程节点） | `load(id) -> Scenario` | 剧本文件(JSON) |
| **GameGraph**（**手写确定性控制器**，非 LangGraph） | 定义并推进阶段、决定当前轮到谁；**按 `reveal_phase` 确定性地发线索、组织投票** —— 原"DM 的活"里**确定性的部分归此，不交给 LLM 拍板**（它看得到真相，但只输出动作、不生成自然语言） | `step(state) -> state'` | 编排层；调度 DMSpeaker/Player、ScenarioLoader |
| **DMSpeaker**（DM 发言 Agent） | **只拿 public-safe context** 生成主持话术：宣布阶段、演 NPC、点名提问、临场圆场；要宣布的线索文本由 GameGraph"递给它"，它**无权看全量真相**。揭真相阶段才显式放开 truth context | `dm_speak(public_safe_ctx, instruction) -> 话术` | LLMRouter(强模型)、VisibilityGate |
| **Player Agent ×N** | 基于自己可见信息发言/答辩/撒谎/推理/投票 | `act(visible_ctx, prompt) -> 发言/投票` | LLMRouter(便宜模型)、MemoryManager、VisibilityGate |
| **VisibilityGate** ⭐ | 算出某参与者此刻能看到的信息集合，物理隔绝别人秘密与上帝视角 | `visible_for(pid, state) -> VisibleContext` | 对局状态、Scenario |
| **MemoryManager** | 维护每人记忆（说过/听到/当前怀疑），长对话摘要防漂移 | `recall(pid)` / `append(pid, ev)` | 对局状态 |
| **LLMRouter** | 按角色路由模型，统一重试/超时/降级，模型无关 | `complete(role, msgs) -> text` | 各家 LLM SDK |
| **LeakDetector** ⭐ | **在线回路**：发言生成后用规则层（关键词 + 别名 + 禁止对象）比对禁止集合，命中则打回重生成。保持便宜+确定，不在实时回路里塞慢/非确定的 LLM-judge；语义/embedding/LLM-judge 检测放离线 eval 车道（见 §8） | `check(pid, utterance, state)` | Scenario、VisibilityGate |

⭐ 两个标星组件 = 自研核心、核心学习点。`VisibilityGate` 事前不给、`LeakDetector` 事后兜底，双保险防"AI 开天眼"。

**最小数据模型（所有组件的共同锚点）**

VisibilityGate / LeakDetector / eval 都围绕这三个结构展开，先把它们钉死（示意，伪 schema）：

```
// InfoItem —— 一条可见性受控的信息（私密剧本/秘密/目标/线索/真相都是它）
InfoItem {
  id: string,            // 唯一 info_id，是 prompt 组装与泄密检测的锚点
  scope: "private" | "public" | "directed" | "omniscient",
  owners: string[],      // scope=private/directed 时，能看到它的参与者 id
  reveal_phase: string,  // 最早可公开的阶段；早于此阶段出现即视为泄密
  text: string,          // 正文
  aliases: string[],     // 别名/同义表达，供 LeakDetector 规则层比对
  tags: string[]
}

// Event —— 对局中发生的一件事
Event {
  id: string,
  type: "utterance" | "clue_release" | "vote" | "phase_change",
  actor: string,         // 参与者 id 或 "DM"/"engine"
  visibility: "public" | "directed" | "private",
  payload: object        // 按 type 定义
}

// GameState —— 引擎单一事实源
GameState {
  phase: string,
  participants: Participant[],
  public_events: Event[],
  private_events: { [pid]: Event[] },
  revealed_info: string[]    // 已揭晓的 info_id 集合
}
```

注：**"线索发早了"本身就是一种泄密** —— `reveal_phase` 让它从模糊约定变成可被 VisibilityGate / eval 断言的不变量。

## 5. 对局流程 + 数据流

**状态机（手写显式状态机）**
```
[开场] → [分配角色] → [自我介绍] → ┌→ [搜证发线索] → [圆桌讨论] ┐
                                   │                            │ (可多轮)
                                   └────────────────────────────┘
                                                 ↓
                                  [投票指认] → [复盘揭真相] → 结束
```

**单个"发言回合"数据流（引擎核心循环）**
```
① GameGraph 决定轮到 Player_3
② VisibilityGate.visible_for(P3) → 只装配：P3私密剧本 + 公开发言 + 发给P3的线索
   （绝不含别人秘密/真相/未公开线索）
③ MemoryManager.recall(P3) → 取 P3 记忆
④ 组装 prompt → LLMRouter.complete("player", …) → P3 发言
⑤ LeakDetector.check → 通过✅ / 越权则打回④重生成♻
⑥ 发言广播进"公开发言历史"，进入所有人下一回合的可见上下文
⑦ GameGraph 推进到下一参与者（若是真人，则经 WebSocket 等其输入）
```

**人机同管道**：真人就是一个特殊参与者；轮到真人时第 ⑦ 步不调 LLM、改等 WebSocket 输入，其余步骤与 AI 完全一致。

## 6. 信息隔离的落地

给每条信息打"可见范围"标签（即 `InfoItem.scope`），AI 的 prompt 只装它该看的。

| 范围 | 例子 | 谁能看到 |
|---|---|---|
| 🔒 私密 private | 角色私密剧本、秘密、目标 | 仅他自己（+ 引擎/DM） |
| 📢 公开 public | 公开发言、DM 公告 | 所有人 |
| 🎯 定向 directed | DM 私发给某人的线索 | 指定参与者 |
| 👁 全知 omniscient | 真相、凶手、所有人秘密 | 仅引擎/DM |

- **VisibilityGate**：挑出范围标签包含该参与者的信息，拼成 `VisibleContext`；玩家 prompt 只用它来拼 → 没看到的无从说起（事前不给）。
- **DM 边界 = 拆角色，而不是靠自律**：全量真相**不进 DMSpeaker 的 prompt**。确定性的"何时进入哪个阶段、发哪条线索、组织投票"由 GameGraph + Scenario 决定（它们看得到真相，但只输出动作、不生成自然语言）；DMSpeaker 只拿 public-safe context + 上层递来的"要宣布的那条线索文本"生成主持话术。**揭真相阶段才显式把 truth context 放开给 DMSpeaker。**
- **LeakDetector（在线回路，事后兜底）**：发言生成后用规则层比对"禁止集合"（按 `info_id` 取别人秘密的关键词/别名、未公开线索、真相词），命中则打回重生成；连续 N 次仍漏 → 安全模板发言。保持便宜+确定；语义/暗示型泄露的逼近放离线 eval（§8）。

Player prompt 结构：
```
[System] 你是<角色>。私密剧本… 秘密… 目标… 规则：只知道下列信息，严禁编造线索。
[Context] 公开发言历史… 发给你的线索… 你的记忆/当前怀疑…
[Instruct] 现在轮到你发言。
```

## 7. 错误处理（铁律：游戏永远不能崩）

| 故障 | 兜底动作 |
|---|---|
| LLM 超时/失败 | 指数退避重试 → 换备用模型 → 安全发言（"我再想想"），保证轮转不卡死 |
| 结构化输出解析失败 | 要求只输出 JSON 重试 → 仍失败用规则兜底（弃权/默认） |
| 泄密/越权 | LeakDetector 打回重生成 → N 次仍漏 → 安全模板替代 |
| AI 出戏拒演 | 检测拒绝话术 → 加强"游戏框架"提示重试 → 仍不行换模型 |
| 人格漂移/前后矛盾 | 记忆中固定"我是谁/我的秘密/关键承诺"为不可变锚点，每次注入 |
| DM 不推进、卡住 | 每阶段设硬上限（最多 N 轮/M 分钟）到点强制推进 |
| 上下文膨胀/超窗 | 触发摘要压缩；单局设调用预算上限，超了告警 |

核心原则：任何单个 AI 失败只降级、不崩盘 —— 宁可让一个 AI 说句废话，也要让整局走完。

## 8. 测试策略

**第 1 层：确定性部分 → TDD，必须稳过（mock LLM）**
- VisibilityGate（最重要）：断言"某玩家可见集合里绝对没有别人秘密/真相"、且 `reveal_phase` 尚早的线索不出现
- GameGraph：阶段转换正确、不跳阶段、到上限强制推进、按 `reveal_phase` 发线索
- LeakDetector 规则层：已知泄密样本抓得到、正常发言不误伤
- ScenarioLoader / LLMRouter（重试降级用 mock LLM 测）

**第 2 层：AI 部分 → "录制回放 + 不变量断言" eval 台（真实模型）**
- 跑全 AI 自动对局（无真人），录下整局 transcript
- **断言不变量与结构，绝不断言 transcript 文本一致**（对 LLM 做 golden-text 断言必然脆）：①阶段顺序对、没跳阶段 ②可见性不变量成立 ③每个 AI 守住身份/目标 ④凶手没提前自爆 ⑤投票格式合法
- **离线泄密测量车道**：拿录好的 transcript，用泄密样本集 + embedding/语义相似 + LLM-judge + 人工抽检，统计**发言层**泄密率（这是 §1 验收 #2 第二层的数据来源）。只做统计与回归趋势，**不进实时回路**
- 软指标（像不像人、推理顺不顺、好不好玩）→ 人工打分
- fixture 仅用于**不变量/结构回归**，不做整段文本比对

这个 eval 台是核心学习产出，也是"非确定性玄学"的解药，并直接构成下面的风险优先 spike。

## 9. 实现策略：风险优先 + P0 实现切片

**不按"地基到屋顶"顺序盖**，先用最小代价做一个一次性 spike，验证最致命的三个假设（AI 会不会演/会不会撒谎、推理行不行、延迟卡不卡）：

- spike 形态：`一桌 3 个 AI + 一个 10 分钟能跑完的迷你本`，纯命令行，无前端、无 WebSocket、无漂亮架构。
- spike 目标：亲眼看 AI 演得有没有意思、卡不卡、会不会自爆。
- 好玩 → 信心拉满，按本设计放心做主线；不行 → 只花几天即止损，避免在错误假设上盖三个月楼。

**P0 实现切片（把"设计愿景"落成可执行路线）**
1. Scenario JSON + `InfoItem`/`Event`/`GameState` schema（§4 数据模型）
2. VisibilityGate 单测（含 `reveal_phase` 不变量）
3. 3 个 AI 纯 CLI 自动对局跑通
4. LeakDetector 规则层 + 泄密样本集
5. 录制 transcript + 不变量断言（非 golden-text）
6. 上面全绿后，才做 Node 对局服务 / H5

**分期：按「垂直切片」推进，每期都是一个完整、能交付、不糊弄的里程碑（2026-06-08 重定）**

> 单人 + AI 建造要活下来，靠的是「每个里程碑都端到端可用、范围克制」，而不是把每一层都按终极规格盖完再拼。「认真对待」≠「一上来盖满」，而是「每一版都不糊弄，但范围克制」。

- **Phase 0｜spike（已完成 ✅）**：CLI，3 AI 自动对局跑通。已验证三个最致命假设——AI 肯演 / 会撒谎 / 不自爆、推理可用、延迟 ~6s/次可接受（见 `spike/`、`spike/transcripts/`）。
- **Phase 1｜硬核引擎 + eval 台（地基，先做）**：把 spike 的领域逻辑迁到 `src/engine/`（TypeScript，GameState 驱动；Python spike 保留为存档），落地 GameGraph(确定性控制器) ↔ DMSpeaker(public-safe) 拆分；VisibilityGate 穷尽式确定性测试（含 `reveal_phase` 不变量）；LeakDetector 在线规则层；eval 台跑 K 局自动对局产出**指标数字表** + 录制 transcript。交付物：CLI 全自动对局 + 一张 eval 数字表 + 测试全绿。**先做的理由**：风险最高、最差异化，且接口要先在这里稳下来才好往上盖 UI。**此阶段不引入 WebSocket / 前端。**
- **Phase 2｜人在环 + 第一版真实前端（验证「好不好玩」）**：接 `HumanParticipant`；一个**最小但认真**的 web 前端 + 简单后端端点（聊天式：看发言 / 打字 / 投票）。这一版前端是「真实前端的 v1、会持续长大」，不是抛弃式 demo。传输先用最简单能跑的（HTTP / SSE 即可）；**WebSocket 等确有多人 / 实时需求再上**，别为还不存在的需求提前盖传输层。交付物：作者 + 几位剧本杀朋友各自完整玩一局，收回真实反馈。
- **Phase 3｜打磨成可分享的好产品**：前端体验 / 视觉 / 交互认真打磨；第二个剧本（验证引擎对剧本的解耦）；离线语义泄密车道（embedding / LLM-judge）补全 eval；按反馈迭代。
- **未来**：多人 / 语音 / AI 出本只保留 `MessageChannel` / `Participant` 接口语义，不提前实现复杂通道。

**贯穿全程｜项目叙事**：边做边记（关键决策、transcript 摘录、eval 数字、踩坑），最终汇成 README / writeup —— 这是把「真实项目」翻译成「面试优势」的桥，不留到最后补。

**工作方式｜AI 主导建造 + 作者内化小内核（已确认采用）**：本项目用 AI（Claude Code）主导建造，按真实开发流程推进（spec → plan → 垂直切片 TDD → 评审 → 验证）。但**差异化内核（VisibilityGate、GameGraph / DMSpeaker 拆分、eval 台）刻意保持足够小**——AI 先写出可用版本，作者熟悉后**亲手从零重写这部分来真正内化**。这是「能力证明」目标的兜底：作者必须能为内核的每个决策辩护。大宗管道（前端骨架、传输、SDK 封装、JSON 解析）放手交给 AI。

**「符合真实开发」的操作标准（本项目的质量底线）**：① 每个切片端到端可用、可交付；② 确定性内核 TDD、测试全绿才算完；③ 非确定性部分用 eval 台产出可复现的数字，不靠「看一眼感觉还行」；④ 每次改动走分支 + 评审 + 提交，历史清晰；⑤ 关键决策有记录（ADR / writeup）。

spike 本质就是第 8 节 eval 台的最小版；它已验证通过，正平滑长成 Phase 1 的主线（见上）。

## 10. 关键技术决策记录（ADR）

| 决策 | 选择 | 理由 |
|---|---|---|
| 主线语言 | **TypeScript（Node.js 20+）** | 引擎、对局服务与未来 H5 共用一套语言和类型系统，减少双语言切换；当前所需 LLM API、状态机与 eval 能力不依赖 Python agent 框架。已完成的 Python 实验保留在 `spike/`，不再承载正式功能 |
| 前端语言 | **TypeScript（H5，后续小程序）** | 与主线引擎共享类型和工具链，降低接口漂移 |
| 传输 | **先 HTTP / SSE，WebSocket 后置**（保留 `MessageChannel` 抽象） | Phase 2 的人在环前端先用最简单能跑的 HTTP / SSE；WebSocket（双向、预留多人广播与语音实时；微信小程序对 WS 支持也更好）等确有多人 / 实时需求再上，避免为不存在的需求提前盖传输层（见 §9 修订） |
| 编排框架 | **TypeScript 主链路手写极简状态机；框架实验独立、后置**（2026-06-08 pivot 后确认） | 领域规则（GameState/阶段转移/VisibilityGate/胜负判定）与编排运行时**分离**。当前复杂度下显式状态机更透明、可单测、易内化；只有当 checkpoint / 恢复 / 并发 / 人工中断的实际收益超过依赖成本时，才用独立 spike + ADR 评估框架适配，且不改领域逻辑。信息隔离由代码强制，绝不交给提示词或框架 |
| 是否基于 CrewAI | **否** | CrewAI 内核是"协作、共享上下文"，与剧本杀"对抗、信息不对称、欺骗"相反，会在核心难点上对着干。CrewAI 留给二期"AI 出本"（协作生成）场景 |
| LLM | **模型无关，按角色路由**：玩家→DeepSeek（便宜量大），DM→更强模型 | 控成本 + 保关键质量；"按角色分模型"本身是好的 agent 工程练习 |
| DM 安全边界 | **拆角色**：GameGraph(确定性、看真相但只出动作) + DMSpeaker(只拿 public-safe context 出话术) | 不让同一个 LLM 既握全量真相又做自然语言主持，从架构上消除"DM 开天眼"——这正是本项目核心学习点的具体落地 |

## 11. 风险清单与应对

**🔴 决定项目成不成立**
1. AI 不肯演坏人/不肯撒谎（被训练得诚实无害，凶手可能自爆）→ "游戏框架"系统提示授权欺骗；选放得开的中文角色扮演模型；**最早实测**。
2. 推理体验天花板（LLM 多步演绎、识破谎言不稳定）→ MVP 选简单、短、线索链紧凑的小本，把推理难度压在模型能力内。
3. 隐性信息泄露防不全（AI 推理"快得不自然"、用暗示/同义绕过关键词）→ 闸门(事前) + 规则检测(在线) + 离线语义/LLM-judge 抽检(测量) 三道逼近，**无法 100% 根除**；故 §1 验收 #2 拆成"prompt 组装层硬证明 + 发言层统计达标"两层。

**🟠 体验难受**
4. 延迟（一桌 AI 逐个调 LLM，轮一圈干等）→ 流式输出 + 能并行的阶段并行 + "思考中"状态。**注意**：别为追求严谨把慢的 LLM-judge 塞进在线回路（会加重本风险）——语义检测一律走离线。
5. 成本/上下文膨胀 → 便宜模型打底 + 摘要压缩（注意摘要丢信息影响推理的平衡）。
6. 非确定性导致开发"玄学"→ 早搭录制回放 + 不变量断言 eval 台。

**🟡 被低估**
7. "一个好剧本"是专业手艺，逻辑不自洽则 AI 再强也白搭 → MVP 自写极简原创本（绕开版权、好控难度）。

**⚪ 已知次要**：AIGC 标识/备案（上线才触发）、剧本版权（用原创规避）。

## 12. 未来演进预留

- **AI 玩家：从"脚本角色"进化为"自主社交 agent"**（核心学习纵深，分三级，循序渐进）：
  1. **说话像真人、心里真推理**（spike 已起步）：`think`/`say` 拆分 —— 私有思维链（含怀疑、心智揣测、本轮策略）只回流给 agent 自己当记忆、**永不广播**；公开 `say` 走自然口语、禁旁白。借鉴狼人杀 agent 的"先推理再开口"脚手架与心智理论（§13）。这同时堵住"内心独白/紧张 tell 被广播"这条泄密缝。
  2. **有边界的自主**：从"被流程逼着发言"升级到"在有限动作菜单里自己选"（发言 / 沉默 / 点名质询 / 发起归票 / 请求私聊）。铁律约束：GameGraph 仍掌阶段推进与硬上限，自主只在阶段【内】发生 —— 既要鲜活、又要守住"永不崩 / 不卡阶段"（§7）。这是从"脚本流程"升级到"事件 / 动作驱动循环"的一步。
  3. **私有信道与社交博弈**：AI 主动"拉某人私聊"（MVP 即可做：1 真人 + AI，AI 单独找真人玩家问话/结盟），复用 `MessageChannel` + §6 定向可见。难点正是本项目核心命题 —— **私聊所得绝不能在公开发言里泄露**，泄密面显著放大，是 VisibilityGate / LeakDetector 的硬考。"主导投票 / 带节奏"是上述能力的涌现，不单独实现，但要防一家独大。
- **多人同桌**：`Participant` 抽象已预留；加真人 = 多几个 `HumanParticipant`，引擎层不动；实时同步走 WebSocket 广播。
- **语音实时**：引擎内部恒为文字流转；语音是"贴在外面的一层皮"——`语音→STT→文字引擎→TTS→语音`。`MessageChannel` 设计为可承载音频。注意：语音 + 多 AI 会显著放大延迟风险（含打断/抢话），属后续阶段。
- **AI 自动出本 / 创作助手**：协作生成任务，适合用 CrewAI 这类框架做流水线。
- **AI 新手引导助手**：降低剧本杀新人门槛。
- **合规**：对公众上线前补 算法备案 + AI 生成内容标识（架构已留位）。
